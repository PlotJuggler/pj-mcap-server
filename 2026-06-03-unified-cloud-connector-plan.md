## 1. Summary ‚Äî one codebase, two engagements

We will build **one** self-hosted server + **one** PlotJuggler client codebase that serves MCAP recordings from cloud object storage on demand, and use it to deliver **two** paying engagements:

- **Dexory** (AWS S3, self-hosted single Go binary). Commercial frame from the proposal: **Milestone 1** (PoC, EUR 10k, ~1 month) gated on written client approval ‚Üí **Milestone 2** (hardening + PlotJuggler plugin + WASM browser bonus, EUR 6k, ~1 month).
- **Asensus Surgical** (Google Cloud Storage, deploy on **GCE**). SOW deliverables: cloud connection server, PlotJuggler cloud streaming plugin, GCS auth + file browser, "lazy-loading + prefetch" (satisfied per LOCKED 2 by selection-scoped bulk + **in-memory session reuse**; cross-session/cross-restart persistence is **out of v1** ‚Äî owned by Asensus's separate "Task C" disk cache or renegotiated ‚Äî see ¬ß3.4 and ¬ß8), deployment guide + infra requirements, TLS + compression, metadata search (robot ID / procedure date / operator), MCAP-first with an open path to other formats.

**The thesis.** ~70% of the work ‚Äî wire protocol, catalog, indexer algorithm, session producer/consumer + retain/resume, dashboard/metrics, the entire Qt `client-core`, and the integration harness ‚Äî is backend-agnostic *by construction* and is written exactly **once**. It only ever touches two narrow seams (`BlobStore` for storage, `FormatCodec` for recording format), so it cannot know or care whether the bytes come from S3 or GCS, or whether the file is MCAP or something later. The genuine incremental cost of the *second* engagement is **~8‚Äì12%**: the GCS `BlobStore` impl, the storage-credential factory split, a `{s3,gcs}` CI matrix dimension, the Asensus GCE deploy guide, and one additive client UI mode (a GCS file-hierarchy browser). Per-deployment behaviour is **config + a small factory layer that selects seam implementations ‚Äî never a fork.**

This is a deliberately **minimal-diff** evolution of the already-specced Dexory design. The spec had already isolated almost every storage-touching concern behind one narrow interface (`s3reader.RangeGetter`, ¬ß8.3) and one indexer list loop (`ListObjectsV2`, ¬ß5.2). The wire envelope (¬ß6) carries **zero** storage-specific fields, and `files.s3_key` (¬ß5.1) is already a generic TEXT key that holds `s3://‚Ä¶` or `gs://‚Ä¶` equally. So ¬ß6 (wire) and ¬ß7 (user flow) are **not edited at all**.

**Two issues the review surfaced are addressed head-on, not papered over:**

1. **Client caching.** The design spec lists client-side disk caching as an explicit **non-goal** (¬ß2 line 40, ¬ß12 line 897: "Memory-only via `pj_datastore`; no local disk cache"), and the review confirmed `pj_datastore` has **no disk tier**. **Decision (¬ß8): the v1 client cache is in-memory, session-lifetime reuse only ‚Äî which *matches* the spec, so this is NOT a reversal and adds no disk-cache engineering.** The cost lands contractually, not technically: Asensus's "repeated access needs no re-download" is honored **only within a running PlotJuggler session**; persistence across restarts/users is out of v1 (their separate "Task C" disk cache or a renegotiated SOW line). See ¬ß3.4.
2. **Mosaico reuse data-model mismatch.** "Lift the Mosaico dialog unchanged, swap three seams" collides with a real mismatch: Mosaico keys on a single-string `sequence_name` with per-topic parallel pulls, whereas our session opens **one** multiplexed WS session over a **set** of integer `file_ids[]` with server-side stitching. This plan specifies the reconciliation (¬ß3.3) and downgrades "reused unchanged" to "reused with one defined selection-model adapter."

## 2. Locked decisions (restated concisely)

1. **ONE codebase** serves both engagements; per-deployment behaviour is config/backend selection, never a fork.
2. **Streaming = whole-selection eager-bulk only.** The client commits to `(file_ids[], topic_names[], time_range)` and bulk-downloads exactly that selection as fast as network+storage allow ‚Äî delivered **progressively** (batch-by-batch) so the client shows already-downloaded data while the rest arrives in the background; "streaming" denotes this incremental, non-blocking delivery, **not** time-paced playback. There is **no** viewport-driven random-access prefetch engine. Fast re-open **within a running session** comes from the in-memory client cache (¬ß3.4).
3. **Client cache = in-memory, session-lifetime reuse for BOTH engagements.** Reconstructed `(file, topics, time-range)` data is reused from memory within a running client session so a repeat open avoids re-download. **No disk tier** ‚Äî this matches design-spec ¬ß2/¬ß12 (no reversal). Cross-restart / cross-user persistence is out of v1 (¬ß3.4, ¬ß8).
4. **Storage backend = abstraction:** a `BlobStore` interface over HTTP Range GETs, with an S3 impl and a GCS impl.
5. **Auth = abstraction with two separate seams:** (a) client‚Üîserver (shared bearer token, pluggable); (b) server‚Üîstorage (S3 credentials vs GCS IAM/service-account). Kept strictly independent.
6. **Client UI reuses the Mosaico `toolbox_mosaico` conceptual design** (connect-to-URI + history, browse with metadata, Lua metadata query filter, topic list, time-range slider, multi-topic fetch with progress, settings persistence, datastore ingest). We swap transport (WS + Protobuf) and ingest (raw MCAP records forwarded to PlotJuggler's MessageParser plugins); everything above that line is reusable and overlaps Plan B's `client-core`.

## 3. Unified architecture

### 3.1 Shared Go server core ‚Äî five subsystems, one process, one TCP listener, one TLS cert

Unchanged from spec ¬ß3: **Catalog** (WS RPC + SQLite reads, WAL + single catalog-writer goroutine), **Session** (WS streaming + storage fetcher with producer/consumer split for resume), **Indexer** (background storage poller), **Dashboard** (read-only HTML + Prometheus `/metrics` + `/health`). Pure-Go SQLite (`modernc.org/sqlite`, no cgo). All five subsystems consume storage **only** through `BlobStore` and recordings **only** through `FormatCodec`; no subsystem outside `internal/storage` imports a cloud SDK, and only `internal/format/mcap` imports `mcap-go`.

(Its `GetPreSignedURL` direct-read model is considered and rejected for v1 ‚Äî see design spec ¬ß6.4 "Deferred whole-file fast paths".)

Smaller client seams: **AuthProvider** (bearer-token header + TLS config, persisted per-URI via `SettingsStore`; qtkeychain in the plugin, plain settings in the CLI) and **BackendCapabilities** (`HelloResponse{supports_file_hierarchy, metadata_key_vocabulary}` toggling an *additive* `QTreeWidget` browser mode).

### 3.3 Client ‚Äî `client-core` + Mosaico-derived PJ4 toolbox dialog (with the two corrections)

`client-core` (Qt-aware, **Widgets-free** ‚Äî QtCore/QtNetwork/QtWebSockets) requires **zero** change for either engagement: it speaks the storage-agnostic WS/Protobuf/MCAP wire. Built once, shared by (a) v1 `client-cli` (`McapWriterSink`, proves round-trip equality on both backends), and (b) the deferred PJ4 plugin (`DatastoreSink` + in-memory session reuse), and (c) the Dexory WASM demo (same `client-core` compiled to WASM).

**Genuinely reused unchanged from `toolbox_mosaico`** (verified against source): the Lua metadata query stack (Engine/Parser/AST/assist) ‚Äî which directly delivers Asensus's metadata search by robot ID / procedure date / operator; `computeVisibleSequences`; name/date filters; `server_history` + URI normalization + per-server credential cache; `SettingsStore`; the worker-thread/command-queue/`onTick` async‚ÜíGUI bridge; the three-column splitter UI; table sort + selection re-map; speed sampling; image ontology routing. Rename `mosaico_` ‚Üí `cloud_connector_`.

**Correction A ‚Äî the `file_ids[]` vs `sequence_name` selection-model adapter (this revises the "1:1 reuse" claim).** Mosaico's `SequenceRecord` is keyed by a single `std::string name` and `pullTopicsAsync(sequence_name, ‚Ä¶)` does **per-topic parallel** pulls of **one** sequence. Our wire model opens **one** multiplexed session over a **set** of `file_ids[]` with server-side stitching. The headline Dexory case ‚Äî "3+ consecutive recordings as one session" ‚Äî is N `SequenceRecord`s in Mosaico's model but **one** `OpenFresh` on the wire. The `BackendConnection` seam therefore **must** include a defined adapter: **a multi-file selection is presented to the dialog as one synthetic stitched `SequenceRecord`** (time range = union, topics = union), and `pullTopicsAsync` on that synthetic record maps to a single `OpenFresh{file_ids[], topic_names[], time_range}`. This is dialog-selection-model logic, not a pure transport swap; it is an explicit M2b deliverable. (Where the data genuinely *is* 1:1: file‚Üísynthetic-sequence, flat `tags_effective`‚Üí`user_metadata`, time slider‚Üí`OpenFresh.time_range`, `Progress`/`Eos`‚Üí`pullProgress`/`pullFinished`/`allFetchesComplete`.)

**Correction B ‚Äî the connector forwards raw records; it does NOT decode (this resolves the earlier "net-new decode" concern in the connector's favor).** **Decision (¬ß8):** the cloud connector is a pure **data source**. It hands each raw MCAP message `(topic, schema_name/encoding, payload, log_time)` to PlotJuggler's existing **MessageParser plugin family** (ROS / Protobuf / JSON-Schema) via the host SDK; **decoding into numeric series is the parsers' job, explicitly out of the connector's scope** (proposal ¬ß3.1). So there is **no decode pipeline to build in M2b** ‚Äî only a thin `SessionSink`/`DatastoreSink` that routes raw records into the host's message-parser dispatch. The remaining concrete unknown is small and bounded: confirm the exact PJ4 SDK entry point that accepts a raw `(schema, topic, payload)` message and routes it to the registered parsers (vs. the Arrow bulk path Mosaico used because its wire format already *was* Arrow). This shrinks M2b relative to the original estimate.

**Caveat on "DialogState compiles unchanged":** true for the filter/query/selection fields, but Mosaico's `topic_meta`/`topic_infos` are Arrow-schema-typed; the topic-detail panel data model is **not** zero-edit. Scope the "zero edits" claim to filter/query/selection only.

Dexory-vs-Asensus on the client is **one binary**: GCS-vs-S3 is invisible (storage auth is server-side); flat-vs-hierarchy is a `BackendCapabilities` flag toggling an additive tree view, not a rewrite.

### 3.4 The client cache ‚Äî in-memory, session-lifetime reuse (aligns with the spec)

**No spec edit.** The design spec lists "Local on-disk caching of streamed sessions on the PJ4 client" as a v1 **non-goal** ("Memory-only via `pj_datastore`"; "no local disk cache"), and the review confirmed `pj_datastore` has only an in-memory store with retention-window eviction ‚Äî **no disk tier**. **Decision (¬ß8): the v1 cache is in-memory, session-lifetime reuse only.** This *matches* the spec, so ‚Äî unlike an earlier draft of this plan ‚Äî it is **not** a reversal and carries **no net-new disk-cache engineering.**

What v1 delivers and what it does not:
- **`SessionCache` is an in-memory map** keyed on the normalized `(server_uri, file_ids[], topic_names[], time_range)` tuple. Within one running PlotJuggler/CLI process, re-opening an identical selection is served from memory ‚Äî **no second download, server-side session count == 0** on the repeat open.
- **Exact-tuple match only** (no subset/superset partial reuse ‚Äî consistent with whole-selection bulk, LOCKED 2). **Only COMPLETE downloads are cached;** `Eos{CANCELLED|ERROR}` leaves no entry. Bounded by a memory budget with LRU eviction (no persistent metadata, no on-disk artifact).
- **Not persistent.** A PlotJuggler restart, or a different engineer, re-downloads. So Asensus's SOW line *"repeated access to the same time ranges does not require additional downloads"* is honored **only within a session**, not across restarts/users.

**Residual contractual item (¬ß8):** if Asensus needs cross-restart / cross-user persistence, that is owned by their separate **"Task C" L3 disk cache** (the connector would *integrate* with it once it exists), or the SOW phrasing is confirmed/renegotiated. This plan does **not** build a disk cache.

### 3.5 Deployment ‚Äî GCE confirmed; Cloud Run out for v1

**Baseline (both engagements, CONFIRMED ¬ß8): a single long-lived container / process.** Dexory = self-hosted single Go binary via systemd or Docker. Asensus = a long-lived container on **GCE** (Compute Engine VM or managed instance group), with an instance-attached service account for ADC (no key file) and a persistent disk for `catalog.db`.

**Constraint ‚Äî Cloud Run is OUT for v1** (decision recorded; state it in the Asensus SOW). The server holds four pieces of in-process, in-memory state that Cloud Run's request-scoped, scale-to-zero, horizontally-autoscaled model breaks: (1) the session registry (`subscription_id ‚Üí *Session`); (2) the bounded retain buffer (256 seqs / 64 MB) backing reconnect-resume; (3) the 60 s retain-after-disconnect window + producer/consumer goroutines that survive a WS drop; (4) the always-on background indexer poller (no request hosts it). Single-writer SQLite + in-memory sessions also do not scale horizontally, so **v1 is explicitly single-instance** (SQLite WAL ‚Üí Postgres later if multi-instance is ever needed).

**Why GCE costs Asensus nothing on its stated deliverables:** "lazy-loading + prefetch" is met **client-side within a session** (selection bulk + in-memory reuse, ¬ß3.4), not by server-resident per-view state; cross-session persistence is a separate concern (Task C / wording, ¬ß3.4) and is not a server responsibility either way.

**Defer (scoped paid follow-on, only if Asensus later mandates Cloud Run):** a "stateless mode" persisting session state to Redis/Firestore + the indexer as a Cloud Scheduler job. A meaningful change, not v1.

> **Note for the deployment narrative:** keep "resume" (server-side retain, mid-stream WS drop) distinct from "fast re-open" (client in-memory reuse). The client cache must **not** be presented as a substitute for the spec's server-side resume guarantees; both exist and serve different purposes.

## 5. Concrete milestones

Shape = shared-core milestones (M0, M1a, M1c, M2a, M2b) + thin per-client adapter milestones (M1b GCS, M2c-DEX, M2c-ASEN). Sequencing follows spec ¬ß13 (1a proto ‚Üí 1b server ‚Üí 2 client-core ‚Üí 3 round-trip ‚Üí 4 deferred plugin), refactored into a kill-two-birds shape.

> **Effort vs calendar (must be reconciled with the user ‚Äî see ¬ß8).** The proposal commits ~1 month per Dexory milestone. The effort estimates below sum to more than that. Two honest options: (i) the per-month commitment is a *calendar* commitment for *Dexory-funded scope only* and the effort estimates are wall-clock-with-slack to be compressed/parallelized; or (ii) the proposal's durations need revising. The plan must not silently claim both. **The S3-only Dexory M1 path (M0 + M1a-S3 + M1c) is what must fit Dexory's month; M1b (GCS) is Asensus-funded and must not be billed into Dexory's fixed-price M1.** (The 2026-06-03 decisions trimmed M2b ‚Äî in-memory cache instead of a disk cache, and forward-raw instead of in-connector decode ‚Äî which eases, but does not erase, the overrun.)

> **M1 contract-scope reconciliation (must-fix, see ¬ß8).** The proposal (lines 119-120) lists **tag editing** and **intra-file time-range selection** as **OUT of Dexory M1** ("M1 supports whole-file topic filtering; M2 adds intra-file time windows"). **This plan's default is to honor the contract:** `UpdateTags` and intra-file time-range slicing move to **M2** for the Dexory engagement. The engine code can still be *built* early (it is cheap and Asensus may want time-range in its own timeline), but the **Dexory M1 acceptance gate must test whole-file topic filtering only**, and the M1c round-trip matrix's time-range dimension is exercised under Asensus/internal scope, not as a Dexory M1 gate ‚Äî **unless** Dexory agrees in writing to an expanded M1.

## 8. Open items

### Resolved (decided 2026-06-03)

These were the technical open items; they are now fixed and reflected throughout this document:

- **A. Client cache ‚Üí in-memory, session-lifetime reuse only.** No disk tier; matches design-spec ¬ß2/¬ß12 (no reversal); no net-new disk-cache work. Cross-restart/cross-user persistence is out of v1 (‚Üí residual item below). (¬ß2.3, ¬ß3.4)
- **B. Asensus deployment ‚Üí GCE long-lived container; Cloud Run OUT for v1** (paid stateless-mode follow-on only if later mandated). (¬ß3.5)
- **C. GCS access ‚Üí server-owned single service-account / Workload Identity** (ADC on a GCE instance, no key on disk); per-user impersonation is a documented v2 extension. (¬ß3.2 seam 2)
- **D. Metadata source ‚Üí MCAP footer** (`FormatCodec.Extract`‚Üí`tags_embedded`); no object-custom-metadata read, no `ObjectInfo.Metadata` field, no `BlobStore.Head` metadata path. (¬ß3.1 indexer)
- **E. Client ingest ‚Üí forward raw records to PlotJuggler's MessageParser plugins; the connector bundles no decoders.** No decode pipeline in M2b ‚Äî only a thin forward-to-parser sink. (¬ß3.3 Correction B)
- **F. Multi-format depth ‚Üí one `FormatCodec` interface + one MCAP impl + a documented extension path; no plugin system.** (¬ß3.2 seam 4)
