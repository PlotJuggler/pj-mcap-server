What "streaming" means here

In this project, "streaming" is a progressive bulk download, sent as fast as the network and storage allow. It is not real-time playback, and it does not replay messages at their original recording speed. The only reason it is called streaming is that delivery is incremental and the interface never blocks: PlotJuggler shows and plots the part that has already arrived while the rest keeps downloading in the background.

1. Goals and the core idea

Today, engineers who want to inspect a recording first download the entire log file from cloud storage, then open it locally. The files are large, so this is slow and wasteful — most of the time only a few signals over a short window are actually needed.

The Cloud Connector removes that step. A user connects PlotJuggler to a server, browses the available recordings, picks exactly the files, topics, and time range they care about, and the server streams only that selection back. Here "streams" means a fast incremental download, not real-time playback: the data appears in PlotJuggler as it arrives, without ever saving a full file to disk.

Two customers need essentially the same product, differing only in where the files live and how access is authorized:

  • Dexory stores recordings in AWS S3 and self-hosts a single server binary.
  • Asensus Surgical stores recordings in Google Cloud Storage and runs the server on a Google Compute Engine virtual machine. They also need to search recordings by metadata such as robot identifier, procedure date, and operator.
The core idea

About 70% of the work — the network protocol, the catalog, the indexer, the streaming engine, the entire client library, and the test harness — does not depend on the storage provider or the file format. That part is written once. It touches storage through one small interface and recordings through one small interface, so it never needs to know whether the bytes came from S3 or from Google Cloud Storage.

The extra cost of serving the second customer is roughly 8–12%: a Google Cloud Storage backend, a credential setup, an added test dimension, a deployment guide, and one optional file-browser view. Each deployment is configured, never forked into a separate codebase.

The server

The server is one self-contained Go binary listening on one network port. Inside it, a few cooperating parts share that port:

  • Catalog — answers requests to list and search recordings. It reads from a local SQLite database (using a pure-Go driver, so there is no C dependency to compile).
  • Session — does the actual streaming. It reads the needed pieces of each file from storage and sends them to the client in batches.
  • Indexer — runs in the background, periodically scanning storage for new or changed recordings and updating the catalog.
  • Dashboard, health, and metrics — a read-only status page plus standard health and monitoring endpoints.

The client

The client is built around client-core, a C++ library that handles the connection, the protocol, and decompression. It is deliberately free of any user-interface toolkit so the same library can be used in three places without change:

  • a command-line tool used to prove the system reconstructs data correctly;
  • a PlotJuggler 4 plugin (delivered in the second milestone) with a graphical browse-and-select dialog;
  • a browser version for Dexory, compiled to WebAssembly (a preview, not a contractual feature).
Important design choice

The connector does not interpret the contents of messages. It passes each raw message to PlotJuggler's existing message-parser plugins (for ROS, Protobuf, and JSON-Schema), which already know how to turn messages into numeric series. This keeps the connector small and means it ships no format-specific decoders of its own.

4. The two seams that keep it one codebase

"Seam" here means a small, well-defined boundary where one implementation can be swapped for another. The whole "build once, serve two customers" approach rests on a handful of these. The two that matter most:

BlobStore
(storage)		 Reading a byte range from an object, getting its size, and listing objects. Errors are normalized to two kinds: temporary (retry) and permanent (fail). An S3 implementation over the AWS SDK. A Google Cloud Storage implementation. Same interface, same behavior.

(two independent parts)		 (a) Proving the client may talk to the server. (b) Proving the server may read the storage bucket. The two are kept separate on purpose. (a) A shared bearer token. (b) Standard AWS credentials. (a) The same shared bearer token. (b) A server-owned Google identity (Workload Identity), with no key stored on disk.

(recording format)		 Reading a recording's summary, planning which chunks to fetch, and iterating its messages. One MCAP implementation, shared by both. A future format would be a new implementation behind the same interface — there is no plugin system in version 1, by deliberate choice.

A strict rule enforces this separation: only the storage package is allowed to reference a cloud provider's SDK, and only the MCAP package is allowed to reference the MCAP library. A check in the build fails if anything else does. This is what prevents the single codebase from quietly drifting into two.

5. How streaming works (the bulk download)

What this system calls "streaming" is a bounded bulk download, sent as fast as the network and storage allow. It is not paced playback, and it never replays the recording at its original speed. The client commits up front to a precise selection — a set of files, a set of topics, and a time range — and the server plans the whole transfer and sends it. The word "streaming" here means only that the data is delivered progressively: it arrives in batches, and PlotJuggler shows the part that has already arrived while the rest keeps downloading in the background. The interface stays responsive throughout — there is no frozen window waiting for a large transfer to finish, and no attempt to replay the recording at its original speed.

Everything travels over one WebSocket connection. Catalog requests and the data stream share that single connection, wrapped in a Protobuf envelope that says what each message is. A useful property of the protocol: it carries no storage-specific fields at all, which is part of why the storage provider stays invisible to the client.

A full session. The server gives a size estimate before sending, so the client can show an accurate progress bar.

Stitching several recordings into one

A common case is selecting three or more consecutive recordings and viewing them as a single continuous session — one time range, the combined set of topics, one ordered stream. The server does this joining; the client simply asks for the set of files and receives one stream.

Compression makes the download fast

Each batch is compressed before it is sent: the server groups a batch's messages together and compresses the whole group as one block (using ZSTD). Doing this per batch — rather than one message at a time — matters because robotics logs are dominated by many small, repetitive messages (for example a transform topic at a high rate). Compressing them together shrinks them far more than compressing each one alone, and it uses less processing on both ends. The compression is independent of how the source files happened to be stored. Each batch is compressed on its own, which is also what lets a dropped connection resume cleanly from the last completed batch.

Resuming after a dropped connection

If the network drops mid-transfer, the client can reconnect and continue from where it left off, as long as it returns within a short retention window. The server keeps a bounded buffer of recent data for exactly this purpose. If the window has passed, the server says so clearly and the client simply starts the selection over.

stateDiagram-v2
  [*] --> Streaming
  Streaming --> Disconnected: network drop
  Disconnected --> Resumed: reconnect within the retention window
  Resumed --> Streaming: continue from last acknowledged point
  Disconnected --> Expired: window elapsed
  Expired --> Streaming: client restarts the selection
  Streaming --> [*]: End of stream (COMPLETE)
Resume after a drop. The client can also cancel mid-stream at any time; the server stops and discards the buffer.

6. Catalog, search, and indexing

The catalog is a small SQLite database holding one row per recording, plus its topics and its tags. Tags can be edited; an edited tag takes precedence over the value read from the file, through a simple "override wins" view.

The indexer keeps the catalog current. On a schedule (every few minutes by default) it lists storage and detects changes by comparing each object's identity, size, and last-modified time. For new or changed recordings it reads the file's summary and records its topics and embedded metadata.

Where search metadata comes from

Asensus's search by robot identifier, procedure date, and operator is powered by metadata stored inside each MCAP recording (its footer), read during indexing. The system does not rely on storage-level custom metadata. This keeps S3 and Google Cloud Storage behaving identically and avoids an extra storage call.

8. Re-opening without re-downloading

Within a running client session, re-opening the exact same selection is served from memory — no second download, and the server sees no new session. The cache key is the selection itself (server address, files, topics, time range). Only completed downloads are cached; a cancelled or failed one leaves nothing behind.

A deliberate limit

This cache lives in memory only. It speeds up repeated access during one running session. After PlotJuggler is restarted, or for a different engineer, the data is fetched again. Persistent, cross-restart caching is intentionally out of version 1. For Asensus this is the one place the wording of their statement of work needs confirming — long-term caching is meant to be handled by their separate disk-cache task, which this connector can integrate with later.

9. Deployment

Both customers run the server as a single long-lived container or process. Dexory self-hosts it. Asensus runs it on a Google Compute Engine virtual machine, using a machine-attached identity (so no credential file sits on disk) and a persistent disk for the catalog database.

Why not a serverless platform (Cloud Run)

The server holds live state in memory: the set of active sessions, the resume buffer, the short window during which a dropped client can reconnect, and the always-on background indexer. A scale-to-zero, request-scoped serverless platform breaks all four. Version 1 is therefore a single always-on instance. A serverless mode would be a separate, paid follow-on if it is ever required.

11. Testing strategy

Testing is layered, and every layer is tied to a milestone gate. Cross-backend coverage is built in: the same tests run against both an S3 emulator and a Google Cloud Storage emulator, and a failure on either backend blocks the change.

LayerWhat it checksWhen it runs
UnitIndividual pieces in isolation: catalog queries, indexer change detection, format reading, plan building, error normalization, protocol encoding (verified against a shared reference set in both Go and C++).Every change.
ComponentOne language against live local storage emulators: full session lifecycle, cancel, resume, error injection, backpressure, and the in-memory cache — run against both S3 and Google Cloud Storage.Every change.
End-to-endThe real server and the real client together: stream a recording out and read it back, then confirm it matches the original message-for-message. This is the version-1 release gate.Before merging; on both backends.
BenchmarkThroughput, compression cost, behavior under a slow client, and a storage-parity check (Google Cloud Storage within ~10% of S3). Treated as a regression guard, not an open-ended target.Nightly and on release.
The headline correctness check

The release gate is message-level equality, not byte-for-byte file equality. Two valid writers can produce different bytes for the same data (different chunking or ordering), so the system compares the meaningful content: topic, timestamps, payload, and schema. This matches what the design specification already requires.

12. Status and open items

All the technical decisions are settled. What remains is mostly commercial — questions only the customers can answer.

Settled (decided 3 June 2026)

  • Client cache is in-memory and session-lifetime only (matches the specification; no disk caching is built).
  • Asensus deployment is a long-lived Google Compute Engine container; the serverless option is excluded from version 1.
  • Google Cloud Storage access uses a server-owned identity (Workload Identity), with no key on disk. Per-user access is a future extension.
  • Search metadata comes from the MCAP file footer, not from storage-level custom metadata.
  • Message handling: the connector forwards raw messages to PlotJuggler's existing parsers; it bundles no decoders.
  • Format support: one interface with one MCAP implementation and a documented path to add more — no plugin system.

Open — needs the customers

  1. Dexory milestone-1 scope. Tag editing and intra-file time-range selection are outside milestone 1 in the proposal. Confirm they move to milestone 2, or agree (in writing) to expand and possibly re-price milestone 1.
  2. Acceptance wording. The proposal says "byte-for-byte"; the implementation provides message-level equality (what the design specification itself defines). Confirm this satisfies the milestone-1 sign-off.
  3. Timeline versus price. The effort estimates exceed one month per milestone. Confirm whether the one-month figure applies to Dexory-funded scope only, or whether durations and price need revising.
  4. Asensus funding trigger. Asensus's main deliverables share milestone-2 work, which is gated on Dexory's approval. Define how that work is funded and triggered for Asensus independently.
  5. Cache wording for Asensus. Confirm that cross-restart, cross-user caching is owned by Asensus's separate disk-cache task, or renegotiate the relevant statement-of-work line.
  6. Reference machine. Pin the exact CPU, network, and disk for the 200 MB/s throughput gate so it is enforceable.
