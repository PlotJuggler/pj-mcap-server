1. The whole thing in one paragraph

Dexory's scanning robots record data into files that sit in cloud storage — a large online folder (here, an Amazon S3 bucket). Normally, to look at one recording in PlotJuggler (the graphing app), you'd download the whole 5–22 MB file first. This project removes that wait. A small server sits next to the storage and keeps a fast, searchable index (a "catalog") of every recording. A plugin inside PlotJuggler lets you browse that catalog, filter it down to the recordings you want, tick the few signals you care about, and stream just those — piece by piece, while you watch. You never download whole files, and you only pull what you select.

The one-sentence version: the server builds a searchable index of what's in cloud storage so that browsing and filtering millions of recordings is a fast database query (never a scan of the files themselves), and then streams only the slices you pick.

"Streaming" here means progressive download — bytes arrive in batches so you can start looking while the rest comes. It is not real-time playback paced to a clock.

2. The big picture

Three players. Understand these and you understand the system.

The lake holds the files. The server indexes + serves slices. PlotJuggler browses + graphs. One connection between them.

The server is one small program with five jobs. Two of them are the stars of this document:

PartIts job, in plain words
Indexer ★In the background, walks the storage and writes down what's in each recording — its signals, time span, counts, and the labels in its folder path. This is metadata indexing (section 4.1).
Catalog ★Answers "which recordings match these filters?" instantly, by querying that written-down index — never by opening files. This is filtering (sections 4.3–4.4).
SessionDoes the actual download: reads the needed pieces, filters to your signals/time, compresses, and streams them.
StorageThe one part that talks to Amazon S3 (or Google Cloud). Everything else is storage-agnostic.
DashboardOptional operator web page + health/metrics. Off by default; not part of normal use.

3. The journey of one recording, start to finish

One recording's whole life, and what the data looks like at each step. We follow a real Dexory staging recording.

Stage 1 — A file sits in the lake

It's an MCAP file (the standard robot-recording format). Its location in the bucket is a folder path that encodes labels — this is how Dexory organises the lake.

What it looks like (a real Dexory staging key)
customer=dexory/customer_site=nashville/robot=arri-182/source=ros-bags/date=2026-05-19/rosbox_2026-05-19_16-43-46.mcap
   ~22 MB of binary, ~171 signals, ~630,000 messages, ~23 chunks

Stage 2 — The indexer writes down what's in it (once)

The indexer reads only the table of contents at the end of the file (a few KB, not 22 MB) plus the folder path, and records the facts into the catalog database. After this it never re-opens the file to answer questions about it.

What it becomes (catalog rows — see 4.1/4.2 for the full picture)
files row:  message_count=630000  topic_count=171  chunk_count=23  size_bytes=22M
            start_ns=…  end_ns=…  s3_key="customer=dexory/…/rosbox_2026-05-19_16-43-46.mcap"
            fingerprint = (etag, size, last_modified)   — used to detect changes
path labels (parsed): customer=dexory  customer_site=nashville  robot=arri-182  source=ros-bags  date=2026-05-19

Stage 3 — You browse and filter

You connect, and the server answers filter queries from the catalog database instantly — no file reads, no bucket reads. (This is the part that scales to millions; sections 4.3–4.4.)

Example: "recordings that contain /tf"
→ a SQL query on the indexed catalog → a page of ~200 matching recordings, in milliseconds

Stage 4 — You pick signals (and maybe a time window)

You tick a few signals. The server computes which file pieces it needs and returns an estimate before downloading anything.

What you send → what you get back
you →  OpenSession(this file, topics=[/tf, /odom], whole time range)
← server  approximate_messages, estimated_chunk_bytes  (only your 2 signals)

Stage 5 — The server streams it in small compressed pieces

It reads the needed pieces, keeps only your signals, packs them into ~512 KB batches, compresses each, and sends them. See 4.7.

Stage 6 — PlotJuggler rebuilds it and graphs

The plugin decompresses each batch, checks it's intact, and hands the messages to PlotJuggler. The rebuilt data is identical in meaning to the original (same signals, timestamps, values).

4. Key questions, answered in detail

4.1 ★ What metadata gets indexed, and how

Short answer: for every recording, the indexer records three kinds of metadata into the catalog: (1) derived facts computed from the file's table of contents (counts, time span, size), (2) the folder-path labels (for Dexory: customer, site, robot, source, date), and (3) tags (key/value labels — auto-read from inside the file if present, plus any a person adds). It does this by reading only a few kilobytes per file, not the whole file.

The three kinds of metadata, in detail

(1) Derived facts — computed from the file's table of contents. Every recording gets these 8 derived keys:

s3_key · size_bytes · message_count · topic_count · chunk_count · start_ns · end_ns · duration_ns

(2) Folder-path labels — Dexory stores files in a "Hive-partitioned" layout: each folder is a key=value pair. A generic parser splits the path into labels.

A real Dexory key, parsed into labels
customer=dexory/customer_site=nashville/robot=arri-182/source=ros-bags/date=2026-05-19/rosbox_2026-05-19_16-43-46.mcap
        │              │                    │              │               │            │
   customer=dexory  customer_site=    robot=arri-182   source=         date=        filename=
                    nashville                          ros-bags        2026-05-19   rosbox_2026-05-19_16-43-46.mcap

source=ros-bags = the full recording (~22 MB);  source=low-fat-bags = a slimmed copy (~5.5 MB)

In the file list, this long key is shown as values only, with the date column split out: dexory / nashville / arri-182 / ros-bags / rosbox_2026-05-19_16-43-46.mcap.

(3) Tags and per-signal rows. "Tags" are key/value labels. They can be baked into the file (MCAP "metadata records") or added by a person later. Real Dexory recordings carry no baked-in tags, so unless someone adds them, the tag set is empty — the filterable metadata for Dexory is the derived facts + path labels (+ any tags a person chooses to add). Separately, the indexer writes one row per signal in the file (signal name, its data type, and its message count).

How "indexing" actually happens (and why it's cheap)

  1. The indexer lists the bucket (or a chosen folder of it) to get the file paths.
  2. For each new or changed file it reads only the table of contents — about 3 small reads, regardless of whether the file is 5 MB or 22 MB — and parses the path.
  3. It writes the rows above into the catalog database.
  4. It records a fingerprint per file: (etag, size, last-modified). On the next pass, if all three still match, the file is skipped — no read at all. So a restart over an already-indexed lake re-reads zero files. (On Google Cloud the "etag" is the file's generation number; same idea.)
Real example — the indexer's own log, cold vs. warm
cold (first time):    scanned=104  new=104  reindexed=0  unchanged=0   — reads every file's table of contents
warm (restart):       scanned=104  new=0    reindexed=0  unchanged=104  — reads NONE; ready in milliseconds
one file overwritten: scanned=104  new=0    reindexed=1  unchanged=103  — re-reads just the changed one

4.2 ★ What is in the database

Short answer: a single SQLite file (no database server to run) that holds the metadata above — and never the recordings' actual data. Four tables, one helper view, plus a failures log. It's the searchable index that makes filtering fast.
TableOne row is…Holds
filesone recordingthe full key, the fingerprint, size, time span, total messages, chunk count
topicsone signal in one filesignal name, its data type, its message count
tags_embeddeda label baked into the filekey → value (Dexory: none)
tags_overridea label a person editedkey → value, or "hide this label"
tags_effective(a view — computed, not stored)the merged labels: a person's edit always wins over a baked-in one
indexer_failuresa file that wouldn't readwhich file, when, the error
Why two label tables + a view? A person's edits are never lost. The file's own labels are rewritten on every re-read; a person's edits live in a separate table the re-read never touches. The view merges them with "human edit wins." That's how a label you set survives a forced re-scan.
Real example — the rows for a Dexory recording
-- files (one row)
id=58291
s3_key="customer=dexory/customer_site=nashville/robot=arri-182/source=ros-bags/date=2026-05-19/rosbox_2026-05-19_16-43-46.mcap"
message_count=630000  topic_count=171  chunk_count=23  size_bytes=22020096
start_ns=…  end_ns=…   fingerprint=(etag,size,last_modified)

-- topics (171 rows; a few shown) — one per signal
file_id=58291  "/tf"     type="tf2_msgs/msg/TFMessage"     count=…
file_id=58291  "/odom"   type="nav_msgs/msg/Odometry"      count=…
file_id=58291  "/scan"   type="sensor_msgs/msg/LaserScan"  count=…

-- tags_embedded: none (Dexory recordings carry no baked-in tags)
-- tags_override: none until a person adds one, e.g. after "Edit Tags…":
tags_override:  file_id=58291  key="reviewed"  value="yes"  updated_at=…

Two honest details if pushed: the database lives at /tmp/pj-cloud-catalog.db by default (configurable); a column reserved to cache the table of contents is deliberately left empty (rebuilt on demand). All writes go through a single worker, so reads never block.

4.3 ★ How filtering works (with Dexory examples)

Short answer: there are two kinds. (a) Browse filtering narrows the list of recordings — done entirely in the database with one SQL query that uses indexes; it never opens a file. (b) Download filtering narrows the data inside a recording to the signals and time window you chose. Both are covered below; the scale story for (a) is section 4.4.

(a) Browse filtering — narrowing the list

The server can filter the recording list by exactly these things, and combines them with AND:

FilterMeansIndex that serves it
recorded betweenrecordings whose time span overlaps a windowidx_files_time
has any of these signalsrecordings that contain at least one named signaltopics (file_id,name)
has all of these tagsmatches every key (or key=value) conditiontags PKs + idx_*_kv
has any of these tagsmatches at least one tag conditiontags PKs + idx_*_kv
Real example — "recordings that contain /tf", as actual SQL
you →  ListFiles( has any signal: "/tf" )

the server runs (against the catalog database only — no file or bucket access):
  SELECT f.id, f.s3_key, f.size_bytes, f.start_time_ns, f.end_time_ns,
         f.message_count, f.chunk_count,
         (SELECT COUNT(*) FROM topics t WHERE t.file_id = f.id) AS topic_count
  FROM files f
  WHERE EXISTS (SELECT 1 FROM topics t WHERE t.file_id = f.id AND t.name IN ('/tf'))
  ORDER BY f.id ASC
  LIMIT 201;
← server  a page of matching recordings (each with its labels attached)
Important nuance about Dexory's path labels (get this right). The server's three filter dimensions are time, signal, and tags. The Dexory path labels (customer, customer_site, robot, source, date) are not, today, one of those server filter dimensions. They are handled two ways instead:
  1. Scope at index time. The catalog is pointed at a folder of the lake by a path prefix (e.g. customer=dexory/customer_site=nashville/robot=arri-182/…/date=2026-05-19/). So a whole customer / site / robot / date is selected by what gets indexed.
  2. Parsed for the file list. The full path is stored on every row (the s3_key derived key), and the plugin parses it into the customer/site/robot/source/date columns you see and sort in the list.
To filter the whole lake by, say, robot=arri-182 as a fast server query, those path labels would be promoted into indexed tags at index time (the indexer already reads the path — this is a small, well-understood addition). Tags are a server filter dimension, so once promoted they would filter at full scale exactly like any tag. Be precise on this: the engine filters by time/signal/tag today; path-label query-time filtering is a straightforward extension, not a built-in yet.
Filtering by a tag (the dimension that already scales) — with a Dexory label
you →  ListFiles( has all tags: robot = "arri-182" )   (once robot is indexed as a tag)

  SELECT … FROM files f
  WHERE EXISTS (SELECT 1 FROM tags_effective te
                WHERE te.file_id = f.id AND te.key = 'robot' AND te.value = 'arri-182')
  ORDER BY f.id ASC  LIMIT 201;

Paging through the results

Results come back one page at a time — 200 per page by default, 1,000 maximum. The "next page" marker is simply the id of the last recording on the page; the next query says "give me the recordings after that id." This is what keeps paging fast no matter how big the lake is (section 4.4 explains why).

(b) Download filtering — narrowing the data inside a recording

An MCAP file is split into chunks (blocks of messages). To give you just your signals/time:

  1. Skip whole chunks that contain none of your signals, or sit entirely outside your time window — never read from the bucket.
  2. Drop stray messages in a mixed chunk that aren't your signal or are just outside your time edge.
Real example — pick 3 of 6 signals from the fixture recording (nissan_zala, 33,670 messages)
pick signals = [/nissan/gps/duro/imu, /nissan/vehicle_speed, /nissan/vehicle_steering]
  kept:    14904 + 4513 + 4513  =  23930 messages
  dropped: /current_pose, /status_string, /mag  — never sent
  ← server (before sending a byte): approximate_messages = 23930

A note for completeness: some signals are published once at the very start (a map, a fixed value). With "include latched" on, a mid-recording time window still gets that early value — the server finds the last value before the window and sends it first.

4.4 ★ Filtering at 8 million recordings

Short answer: filtering stays fast at 8 million files because it is a SQL query against a pre-built index, paginated — it never opens a file or touches the bucket, and a page costs the same whether the lake has 100 files or 8,000,000. The hard part at that scale is building the index in the first place (the indexer), not searching it.
Filtering reads the pre-built index, not the lake. A page is a jump-and-read of 200 rows — the same cost at 100 or 8,000,000 files.

Why searching stays fast — three reasons

  1. It's pre-indexed. All the work of opening files happened once, in the indexer. A filter is a plain database query against columns that have B-tree indexes (time, signal name, tag key/value, and the recording id). The database jumps to matches instead of scanning.
  2. It never touches the files or the bucket. Browse + filter read only the SQLite index. Zero object-store requests. So file count doesn't affect whether a filter is "expensive" — there's no per-file work at all.
  3. Paging is jump-and-read, not skip-and-count. Each page is WHERE … AND id > last_id ORDER BY id LIMIT 200. Because the id is the table's primary key, the database seeks straight to that position and reads 200 rows forward. It never counts past skipped rows. So page 1 and page 40,000 cost the same, and memory per page is ~200 rows — constant, regardless of 8 million total.
What that means in numbers, at 8 million files
Catalog size:        ~8M file rows + ~tens of millions of signal rows  →  one SQLite file, ~tens of GB
A filtered page:     a primary-key seek + read 200 rows + 200 small label look-ups  →  single-digit ms
Object-store reads:  0  (filtering is pure database; the bucket is not contacted)
Memory per page:     ~200 rows  (it never holds 8M rows in memory)

The honest part: building the index is the real cost

This is the distinction to lead with if challenged — and it's a strength of the design, because the two halves are cleanly separated:

Stays fast at 8M (no work needed)The real cost at 8M (the indexer)
Searching / filtering — pre-indexed SQL, paginated, no file access. Single-digit ms per page at any scale. Restarts re-read zero files (the fingerprint skip). The first full index build. Cataloguing 8M files means listing them (~8,000 list calls, 1,000 keys each) and reading each new file's table of contents (~3 small reads). Today the indexer does this one file at a time (no parallelism). That's fine at the tested scale (~100 files, ~1.7 s) but at 8M a cold build is a long batch.
What was actually tested, and what 8M would need. The largest validated run was 104 recordings: cold index in ~1.7 s, warm restart re-reads 0, and paging/filters tile cleanly with no gaps or duplicates. The serving side (filter + page) is scale-ready by construction — it's indexed SQL with a primary-key cursor and no file access. To make the building side reach 8M in a tolerable window, the known levers are: (1) index files in parallel (a worker pool) instead of one at a time — the biggest win; (2) write many files per database transaction during the first build; (3) process the file listing as it streams instead of holding all 8M paths in memory; (4) avoid re-listing the whole lake every poll (use change notifications or a "modified since" watermark). These are throughput improvements to the background indexer; none of them change the fast, already-scalable filtering path.

4.5 What is cached

Short answer: the big one is the catalog index itself (so files are never re-read). Plus a few smaller caches. The server never caches raw recording bytes.
CacheWhereHoldsHits when…
Catalog indexserver, on diskall the metadata abovea file's fingerprint is unchanged → not re-read
Retain buffer (resume window)server, memoryrecent pieces not yet confirmed receivedyou reconnect mid-download
Summary read helperserver, memorythe table-of-contents bytes during one scanwithin a single file's indexing
Finished downloadyour computer, memoryper-signal counts of a completed fetchyou fetch the exact same selection again → zero network
Signal listyour computer, memoryeach recording's signal listyou re-open a recording you already viewed

The most important one is the catalog: a restart compares each file's fingerprint and skips unchanged files, so it re-reads none of them.

4.6 How many requests are sent

Short answer: two very different counts. On the one connection to PlotJuggler: ~4 question/answer pairs, then a stream of ~15–40 data pieces, a few progress notes and confirmations, one "done". To the cloud bucket: filtering = 0; indexing a file = ~3 reads (0 if already known); downloading = 1 read per chunk.
Real example — browse, then download one recording
on the connection:
  Hello / HelloResponse             (connect)
  ListFiles / ListFilesResponse     (browse + filter — answered from the index)
  GetFile / GetFileResponse         (one recording's signals)
  OpenSession / OpenSessionResponse (start the download + estimate)
  MessageBatch × ~15–40             (the data, ~512 KB pieces)
  Progress × a few                  (~1 per second)
  Ack × a few                       (you: "got everything through piece N")
  Eos                               ("done")

reads to the bucket for the same session:
  browsing/filtering:    0   (pure database)
  planning the download: 0 if warm, ~3 if never indexed
  streaming:             1 per chunk  (e.g. 9 for the fixture file)

If the connection drops mid-download it reconnects and resumes from the last confirmed piece — no gaps, no duplicates — up to 3 tries.

4.7 How chunking works

Short answer: an MCAP file is built from chunks (blocks of messages) plus a table of contents at the end. The server reads only the table of contents to index and plan (cheap), then reads the chunks you need, re-packs the messages into even ~512 KB batches, compresses each, and sends those. Chunks are how it's stored; batches are how it travels.

The table-of-contents trick is the key efficiency win and worth stating: a recording can be huge, but its table of contents is tiny. The server reads only that — about 3 small reads regardless of file size — to learn every signal, the time span, and the totals. The naive way (reading bits all over the file) was measured at over 3 minutes per file on real cloud storage; the table-of-contents way takes about a second.

Real example — the fixture recording end to end
9 chunks  →  server reads the table of contents (~3 reads) + the 9 chunks (9 reads)
33,670 messages re-packed into ~15–40 batches of ~512 KB each, each compressed as one block
PlotJuggler decompresses each → checks the size matches → decodes → 33,670 messages graphed
the rebuilt data equals the original in meaning (signal, time, value) — checked automatically

5. How it compares to the original plan

Same big shape (one server, one connection, browse + stream, index in a small database). A handful of specifics changed, each for a real reason found while building.

7. Plain-English glossary

TermIn plain words
Lake / bucketA large online folder in Amazon S3 (or Google Cloud) where the recordings live.

MCAPThe standard robot-recording file format. A binder of messages with a table of contents at the back.

MetadataFacts about a recording (its signals, time span, counts, path labels, tags) — not the recorded data itself.

CatalogThe searchable index — the small SQLite database of metadata.
