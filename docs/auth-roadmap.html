User authentication: where it should go in production

A plain-language sketch of how logging in and access control should evolve for the MCAP Cloud server. No prior security expertise assumed.

The MCAP Cloud server streams recordings to clients (the PlotJuggler app and a command-line tool) over a single network connection. Right now it has a very simple lock on the front door. This page explains what that lock is, why it isn’t enough for a real deployment, and a staged plan to make it production-grade — without rebuilding the system.

First: two different questions

Security here comes down to two separate questions. People often lump them together, but keeping them apart makes the whole plan clearer.

Authentication

Who are you?

Proving your identity at the door — like showing an ID badge.

Authorization

What may you do?

Which rooms your badge opens once you’re inside — and whether you can only look, or also make changes.

A complete system needs both. As we’ll see, the first is the easier half to add; the second is where most of the real work (and value) is.

Where we are today: one shared key

Today the server is protected by a single shared secret token — think of it as one house key that everybody copies. A client presents the key when it connects; if it matches, they’re in. The door is fail‑closed: the server refuses to even start unless a key is set (you have to deliberately turn the lock off for local testing).

The shared-key problem. One key for everyone is fine for a small, trusted team, but it has three weaknesses:
  • No identity. Everyone looks the same at the door, so you can’t tell who did what.
  • No boundaries. Anyone with the key can see and download everything.
  • All-or-nothing changes. If one person leaves or the key leaks, you have to change the lock for everybody at once.

A convenient head start

Two design choices already make the upgrade path smooth. First, the “present your key” step in the connection was built as a swap-in point — we can replace the simple key check with something smarter without touching how data is streamed. Second, every recording in the catalog is already labelled by customer (and site, robot, and so on). That customer label is the natural line to draw when deciding who is allowed to see what.

The plan, in stages

Each stage stands on its own and adds real value, so this can be rolled out gradually rather than as one big change.

  1. Personal keys instead of one shared key

    Give each person or program its own API key (a personal keycard) that can be issued, expired, and switched off individually. Now you know who’s connecting, you can revoke one card without disturbing anyone else, and you can mark some cards “read-only” so they can browse and download but not edit labels. No change to how clients connect.

  2. Boundaries by customer

    Attach to each key a list of which customers’ data it may see. The server then filters everything — searches, listings, downloads — to just that data, and it enforces this itself rather than trusting the client. This is what lets a single server safely serve several customers, each seeing only their own recordings.

  3. “Log in with your company account” for people

    For human users of the app, add single sign-on — the familiar “Log in with Google / your work account” flow. People no longer hold a long-lived secret; they log in and the app receives a short-lived pass that expires on its own. Personal API keys stay available for automated tools and scripts.

  4. Operational hardening

    Keep an audit log of who accessed or downloaded which recordings (important when the data belongs to customers), store secrets in a proper secrets manager, slow down repeated wrong-key attempts, and always run over an encrypted connection so keys and data are never sent in the clear.

Two rules to hold onto

  • The server decides, not the client. Whatever a client asks for, the server independently checks that it’s allowed — a client can never grant itself access it doesn’t have.
  • Keep the side door internal. There is a separate internal channel used only for editing labels; it is meant to live entirely inside the server’s own machine and must never be reachable from the network. The main connection stays the one and only front door.

Questions worth settling early

  • Who are the users — people using the app, automated tools, or both? (Decides whether single sign-on is needed at all.)
  • One shared server for several customers, or a separate server per customer? (Decides how important the “boundaries by customer” stage is.)
  • Which company login provider (Google Workspace, Okta, Microsoft…)?
  • Do we need a record of who downloaded what? (For customer data, almost certainly yes — and it’s cheaper to design in early.)
